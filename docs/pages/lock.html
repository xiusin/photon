RAII Guard 自动解锁
LockGuard · 类型安全的 RAII 守卫 · 防止死锁

LockGuard 是 LockManager 锁的 RAII 守卫，当释放时（通过 defer 或手动），锁会自动释放。线程安全的双重释放防护，防止意外解锁。

API说明
LockGuardRAII 守卫，释放时自动释放锁，线程安全双重释放防护
lock_guard(key)获取锁并返回 RAII 守卫
try_lock_guard(key)非阻塞尝试获取锁，返回 ?LockGuard
lock_guard_with_timeout(key, timeout_ms)带超时获取锁，返回 !?LockGuard
v
import photon.locking mut lm := locking.new_lock_manager() // Pattern 1: Guard with defer (recommended / 推荐模式) mut guard := lm.lock_guard('user:123')! defer { guard.release() } // Do work under lock — even if panic, defer ensures unlock // 在锁保护下工作 — 即使 panic，defer 也会确保解锁 // Pattern 2: Try-lock guard / 非阻塞尝试获取 if guard := lm.try_lock_guard('user:456') { defer { guard.release() } // Lock acquired / 获取锁成功 } else { // Lock not available / 锁不可用 } // Pattern 3: Lock with timeout / 带超时的锁 if guard := lm.lock_guard_with_timeout('user:789', 5000)! { defer { guard.release() } // Lock acquired within 5s / 5 秒内获取锁成功 }

零心智成本锁 API
with_lock · 框架保证加锁/解锁 · 用户无需关心

with_lock() 在持有锁期间执行回调，完成后保证释放锁。零心智成本：用户永远不需要考虑加锁/解锁。

API说明
with_lock(key, callback)持有锁期间执行回调，保证完成后释放锁
with_distributed_lock_guard(key, ttl_ms, callback)分布式锁版本的零心智成本 API
v
import photon.locking mut lm := locking.new_lock_manager() // Zero-cognitive-load API: framework handles lock/unlock completely // 零心智成本 API：框架完全处理加锁/解锁 lm.with_lock('inventory:update', fn () ! { // Do work under lock — framework guarantees lock/unlock // 在锁保护下工作 — 框架保证加锁/解锁 deduct_inventory()! })! // Distributed lock with auto-release / 分布式锁自动释放 lm.set_distributed_lock(redis_dlock) lm.with_distributed_lock_guard('order:pay', 5000, fn () ! { process_payment()! })! // Spring equivalent: TransactionTemplate (auto-rollback on exception) // Rust equivalent: MutexGuard (RAII, compiler-guaranteed Drop)
🚀 生产实践
  • 分布式锁必须设置 TTL，防止死锁（进程崩溃后锁自动释放）
  • 长事务使用 renew() 续期，防止业务未完成锁已过期
  • 使用 LockGuard + defer 或 with_lock() 确保锁一定释放，零心智成本，防止死锁
  • 本地锁适用于单进程内的资源保护，跨进程/跨节点使用分布式锁
  • 锁的粒度越细越好：order:123 优于 orders

SessionLockManager
Session 级并发锁 · RAII Guard · 零心智成本

SessionLockManager 复用 locking.LockManager，为每个 Session ID 提供独立的并发锁。不同 Session ID 的操作互不阻塞，同一 Session ID 的操作串行化，确保数据一致性。

核心 API

API说明
new_session_lock_manager(config)创建 Session 锁管理器 / Create session lock manager
lock_session(session_id)获取 Session 锁（通过 acquire）/ Acquire session lock via acquire()
unlock_session(session_id)释放 Session 锁 / Release session lock
acquire(session_id)获取锁并返回 SessionLockGuard / Acquire lock and return RAII guard
release(session_id)释放锁（通常由 Guard 的 defer 自动调用）/ Release lock (auto-called by Guard's defer)
with_session_lock(session_id, callback)零心智成本回调 API / Zero-cognitive-load callback API

SessionLockGuard RAII Guard

SessionLockGuard 是 Session 锁的 RAII 守卫，使用 defer { guard.release() } 模式确保锁一定释放。即使请求 panic，defer 保证锁释放。线程安全的双重释放防护。

ShardedMemorySessionStore 分片 GC
16 分片独立 GC · 减少锁竞争 · 零拷贝读取

ShardedMemorySessionStore 是分片内存 Session 存储，16 个分片各持独立 RwMutex，GC 按分片扫描，避免全局写锁阻塞所有读写操作。读取返回引用而非 clone（零拷贝），安全性由 SessionLock 保证。

API说明
new_sharded_memory_session_store()创建分片 Session 存储（自动启动 GC）/ Create sharded store (auto-starts GC)
read(session_id)零拷贝读取 / Zero-copy read
write(session_id, data, ttl)写入到对应分片 / Write to corresponding shard
destroy(session_id)从分片移除 / Remove from shard
gc(max_age_seconds)按分片过期清理 / Per-shard expiry cleanup
shard_stats()各分片统计信息 / Per-shard statistics
close()停止 GC 协程 / Stop GC goroutine
v
import photon.web mut store := web.new_sharded_memory_session_store() // 不同 session_id 映射到不同分片，互不阻塞 // Different session_ids map to different shards, no blocking store.write('sess-abc', {'user_id': '42'}, 1800)! data := store.read('sess-abc')! // 零拷贝 / Zero-copy // GC 按分片扫描，每次只锁定一个分片 // GC scans per-shard, locking only one shard at a time store.gc(1800)! // 监控各分片 / Monitor shards for stat in store.shard_stats() { println('Shard ${stat.index}: ${stat.count} sessions') } store.close() // 停止 GC / Stop GC

AfterCompletionRegistry 冻结模式
on/freeze/fire · 冻结后无锁快速路径 · finally 语义

AfterCompletionRegistry 管理请求后置回调的注册与触发，参考 HttpKernel 的 on/freeze 模式。冻结后 fire() 无锁读取不可变快照，达到最大性能。

API说明
new_after_completion_registry()创建后置回调注册表 / Create after-completion registry
on(callback, order)注册回调，返回唯一 ID / Register callback, returns unique ID
off(id)移除指定 ID 的回调 / Remove callback by ID
freeze()冻结回调列表为不可变快照 / Freeze callback list as immutable snapshot
unfreeze()解冻，允许修改 / Unfreeze for modifications
fire(ctx, err)触发所有回调（finally 语义）/ Fire all callbacks (finally semantics)
is_frozen()是否已冻结 / Whether frozen
callback_count()回调数量 / Number of callbacks
v
import photon.web mut registry := web.new_after_completion_registry() // 注册回调（order 越小越先执行）/ Register callbacks (lower order executes first) registry.on(fn (ctx voidptr, err ?string) { println('Cleanup resources / 清理资源') }, 10) registry.on(fn (ctx voidptr, err ?string) { println('Log request / 记录请求日志') }, 20) // 冻结后 fire() 无锁快速路径 / Lock-free fast path after freeze registry.freeze() // 每个请求结束时触发 / Fire at end of each request registry.fire(ctx, none) // 正常完成 / Normal completion registry.fire(ctx, 'db error') // 异常完成 / Error completion // 应用关闭时解冻 / Unfreeze at application shutdown registry.unfreeze()
💡 冻结优化：freeze() 后 fire() 直接读取不可变快照，无需加锁。frozen 标志由 freeze() 在写锁下设置，一旦为 true 不会再变回 false（unfreeze() 仅在应用关闭时调用），因此无需原子操作即可安全读取。/ Freeze optimization: After freeze(), fire() reads the immutable snapshot directly without locking. The frozen flag is set under write lock and never goes back to false, so it can be safely read without atomics.

Lock 锁设计说明
Lock Design — SessionLockManager · ShardedMemorySessionStore · AfterCompletionRegistry

SessionLockManager 锁设计

SessionLockManager 复用 locking.LockManager，每个 Session ID 对应独立的 LocalMutex（key = session:${session_id}）。不同 Session ID 的并发操作互不阻塞，同一 Session ID 的操作串行化。

场景锁行为
不同 session_id 并发各自独立锁，互不阻塞 / Independent locks, no blocking
同一 session_id 并发串行化，先到先得 / Serialized, first-come-first-served
acquire 超时返回错误，不阻塞 / Returns error, no blocking

ShardedMemorySessionStore 分片 GC 锁设计

16 分片各持独立 RwMutex，GC 按分片逐个扫描，每次只锁定一个分片，其他分片的读写操作不受阻塞。分片间释放锁，允许其他分片并发操作。

v
// GC 按分片扫描，每次只锁定一个分片 // GC scans per-shard, locking only one shard at a time for i in 0 .. s.shards.len { shard.mu.lock() // ... 清理过期 session / Clean expired sessions shard.mu.unlock() // 分片间释放锁，允许其他分片并发操作 // Release lock between shards, allowing concurrent operations }

AfterCompletionRegistry 冻结模式锁设计

freeze() 后回调列表为不可变快照，fire() 直接读取快照无需加锁（无锁快速路径）。未冻结时 fire() 使用 rlock 克隆后释放锁（慢速路径）。on()/off() 使用 lock 保护写操作。

操作冻结前冻结后
fire()rlock → clone → runlock → 遍历 / Slow path直接读快照，无锁 / Lock-free fast path
on()lock → 添加 → unlock / Write locklock → 添加 → unlock（同步更新快照）/ Write lock + sync snapshot
off()lock → 移除 → unlock / Write locklock → 移除 + 同步快照 → unlock / Write lock + sync snapshot
💡 设计原则：读多写少场景使用 RwMutex（rlock 并行读），写少场景使用 freeze 模式（冻结后无锁读取）。SessionLockManager 的每 Session ID 独立锁策略确保不同用户的操作互不阻塞。/ Design principle: Use RwMutex for read-heavy scenarios (parallel reads via rlock), use freeze pattern for rare-write scenarios (lock-free reads after freeze). SessionLockManager's per-session-ID independent lock strategy ensures different users' operations don't block each other.
