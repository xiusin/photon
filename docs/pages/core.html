IoC 容器
Container · 控制反转的核心实现

Container 是 Photon IoC 的底层引擎，管理所有 Bean 的注册、实例化和依赖解析。推荐使用上层的 ApplicationContext，但 Container 也可独立使用。

基本用法

v
import photon.core // 1. 创建容器 mut container := core.new_container() // 2. 注册 Bean 定义 container.register(core.BeanDefinition{ type_name: 'UserService' scope: .singleton // singleton | prototype | request dependencies: [ core.Dependency{ field_name: 'user_repo', type_name: 'UserRepository', is_required: true } ] }) // 3. 解析 Bean（自动创建实例 + 注入依赖） instance := container.resolve('UserService')! // 4. 检查 Bean 是否存在 exists := container.has('UserService') // true // 5. 按限定符解析 cache := container.resolve_by_qualifier('redis-cache')!

🚀 生产实践
  • 优先使用 ApplicationContext 而非裸 Container，前者整合了生命周期、事件、环境
  • Singleton Bean 的创建顺序由 init_order 控制，确保数据库连接池在 DAO 之前初始化
  • 使用 is_primary = true 解决同类型多实现的歧义，而非到处写 qualifier

依赖注入 (DI)
编译期自动注入 · 零反射开销

Photon 的 DI 通过 V 语言的 comptime $for 在编译期扫描 @[autowired] 注解并生成注入代码，运行时零反射开销。

字段注入（推荐）

v
@[component] pub struct UserService { @[autowired] user_repo &UserRepository // 必需依赖 — 解析失败会报错 @[autowired] cache &CacheService // 必需依赖 } // Spring 等价： // @Service // public class UserService { // @Autowired UserRepository userRepo; // @Autowired CacheService cache; // }

限定符注入（同类型多实现）

v
// 场景：存在多个 Cache 实现（MemoryCache, RedisCache） @[component] @[qualifier('memory-cache')] pub struct MemoryCacheService { /* ... */ } @[component] @[qualifier('redis-cache')] pub struct RedisCacheService { /* ... */ } @[component] pub struct OrderService { @[autowired] @[qualifier('redis-cache')] cache &CacheService // 明确指定使用 RedisCache } // Spring 等价：@Qualifier("redis-cache") // Laravel 等价：$app->when(OrderService::class)->needs(Cache::class)->give(RedisCache::class)

可选注入

v
@[component] pub struct NotificationService { @[autowired] mailer &MailerService // 必需 // 可选依赖 — 容器中没有时不报错，保持 nil slack_notifier ?&SlackService } // Spring 等价：@Autowired(required = false)

集合注入（注入所有实现）

v
// 定义接口 pub interface EventHandler { handle(event string) ! } // 多个实现 @[component] pub struct EmailHandler { /* ... */ } @[component] pub struct SmsHandler { /* ... */ } @[component] pub struct PushHandler { /* ... */ } // 注入所有 EventHandler 实现 @[component] pub struct EventDispatcher { @[autowired] handlers &[]EventHandler // 自动注入所有实现 } // Spring 等价：@Autowired List<EventHandler> handlers; // Laravel 等价：$app->tagged('event-handler')

DeferredProvider（延迟提供者）

v
@[component] pub struct ExpensiveService { @[autowired] heavy_provider core.DeferredProvider // 首次 .get() 才解析 } pub fn (s &ExpensiveService) do_work() ! { // 只有在真正需要时才解析 HeavyBean heavy := s.heavy_provider.get()! // ... } // Spring 等价：ObjectProvider<HeavyBean> // Laravel 等价：Container::lazy()
⚠️ 循环依赖检测：Photon 在编译期检测循环依赖并报错。如果 A 依赖 B，B 又依赖 A，编译时会收到明确的错误信息。解决方案：使用 DeferredProvider 打破循环。

ServiceBinding（Laravel 风格绑定）

v
// 注册绑定（类似 Laravel 的 Container::bind / Container::singleton） mut registry := core.new_binding_registry() // 单例绑定 registry.bind_singleton('CacheService', fn () !voidptr { return unsafe { &RedisCache{} } }) // 每次新建绑定 registry.bind('ShoppingCart', fn () !voidptr { return unsafe { &ShoppingCart{} } })
💡 DI vs Service Locator：DI 是控制反转（依赖被推入对象），Service Locator 是服务拉取（对象主动查找依赖）。Photon 推荐 DI，但 Service Locator 在以下场景很有用：非托管代码、桥接遗留系统、动态服务选择。

全局服务定位器

Photon 提供全局 ServiceLocator API，在应用启动时引导一次后，即可在任意位置（无需传递 Context）进行类型安全的服务查找。

方法返回值说明
locate_service[T]()!&T类型安全全局服务查找
locate_service_by_name(name)!voidptr按名称全局查找
has_global_service(name)bool检查全局服务是否存在
set_global_service_locator(sl)—引导全局定位器（启动时调用一次）
v
import photon.core // 1. 引导：启动时设置全局 ServiceLocator（仅一次） mut app := core.new_application_context() app.refresh()! mut locator := core.new_service_locator(app) core.set_global_service_locator(locator) // 2. 在任意代码中使用 — 无需传递 Context svc := core.locate_service[UserService]()! svc.do_work() // 3. 检查存在性 if core.has_global_service('PaymentGateway') { gateway := core.locate_service_by_name('PaymentGateway')! } // Spring 等价：static ApplicationContext.getBean(MyService.class) // Laravel 等价：app(MyService::class)
⚠️ 全局定位器使用注意：全局 ServiceLocator 适合在非托管代码中桥接访问，但不应滥用。在 DI 容器管理的 Bean 中，仍应优先使用 @[autowired] 注入。

FactoryBean
编程式 Bean 创建 — Spring FactoryBean 等价

当 Bean 的创建需要复杂逻辑（如读取配置、建立连接、条件选择实现）时，使用 FactoryBean 将创建逻辑委托给工厂对象。

v
// 1. 实现 FactoryBean 接口 @[component] pub struct DatabaseConnectionFactory { @[autowired] config &DatabaseConfig } // create() — 生产实际的 Bean 实例 pub fn (f &DatabaseConnectionFactory) create() !voidptr { conn := new_connection(f.config.host, f.config.port)! return unsafe { conn } } // bean_type() — 声明工厂产出的类型名 pub fn (f &DatabaseConnectionFactory) bean_type() string { return 'DatabaseConnection' } // is_singleton() — 工厂产出是否为单例 pub fn (f &DatabaseConnectionFactory) is_singleton() bool { return true } // 2. 注册到容器 mut ctx := core.new_application_context() factory_def := core.new_factory_bean_definition( 'DatabaseConnectionFactory', &DatabaseConnectionFactory{} ) ctx.container.factory_registry.register(factory_def) // 3. 解析时自动走工厂 conn := ctx.resolve('DatabaseConnection')! // 调用 factory.create() // Spring 等价：implements FactoryBean<Connection> // Laravel 等价：$app->singleton(Connection::class, fn() => new Connection(...))

条件装配
@[conditional_on_*] — Bean 按条件注册

条件装配允许根据环境状态决定 Bean 是否注册，这是 Spring Boot 自动配置的核心机制。

注解条件Spring 等价
@[conditional_on_profile('prod')]仅在指定 Profile 激活时注册@Profile("prod")
@[conditional_on_property('key')]配置键存在时注册@ConditionalOnProperty
@[conditional_on_property('key','val')]配置键等于指定值时注册@ConditionalOnProperty(havingValue="val")
@[conditional_on_bean('Type')]指定 Bean 存在时注册@ConditionalOnBean
@[conditional_on_missing_bean('Type')]指定 Bean 不存在时注册@ConditionalOnMissingBean
@[conditional_on_expression('key==val')]表达式为真时注册@ConditionalOnExpression
@[conditional_on_class('Type')]类型存在时注册@ConditionalOnClass

实战示例

v
// 仅在生产环境启用 Redis 缓存 @[component] @[conditional_on_profile('prod')] @[conditional_on_property('cache.driver', 'redis')] pub struct RedisCacheService { @[value: 'redis.host'] host string @[value: 'redis.port'] port int } // 开发环境用内存缓存（当 RedisCacheService 不存在时注册） @[component] @[conditional_on_missing_bean('RedisCacheService')] pub struct MemoryCacheService { /* ... */ } // 自动配置：当 CacheManager Bean 存在且配置开启时注册 @[auto_configuration] @[conditional_on_bean('CacheManager')] @[conditional_on_property('cache.annotations.enabled', 'true')] pub struct CacheAnnotationConfig { /* ... */ }

Resolve 缓存与冻结
Resolve Cache & Freeze — 高性能 Bean 解析优化

ApplicationContext 内置 resolve_cache 缓存机制，避免重复解析同一 Bean。refresh() 完成后自动调用 freeze_bean_definitions()，冻结后 resolve() 走无锁快速路径，达到最大性能。

Resolve Cache 缓存机制

每次 resolve() 调用优先查询缓存（快速路径），缓存未命中时从 Container 解析并写入缓存。注册/移除 Bean 时自动使缓存失效。

API返回值说明
app.resolve('Type')!voidptr优先查缓存，未命中走 Container 解析 / Cache-first resolve
app.cache_hit_count()int缓存命中次数（诊断用）/ Cache hit count (diagnostic)
app.cache_miss_count()int缓存未命中次数（诊断用）/ Cache miss count (diagnostic)
app.cache_size()int缓存条目数 / Number of cached entries

Freeze / Unfreeze 冻结机制

freeze_bean_definitions() 将当前所有 BeanDefinition 快照为不可变 frozen_bean_defs，后续 resolve() 无锁读取快照。unfreeze_bean_definitions() 在 shutdown() 时自动调用，清空快照和缓存。

API说明
app.freeze_bean_definitions()冻结 BeanDefinition 注册表，resolve() 无锁读取 / Freeze bean definitions for lock-free resolve
app.unfreeze_bean_definitions()解冻，允许修改（shutdown 时自动调用）/ Unfreeze for modifications (auto-called by shutdown)
app.is_frozen_bean_definitions()返回是否已冻结 / Returns whether bean definitions are frozen
v
mut app := core.new_application_context() app.register(core.BeanDefinition{type_name: 'UserService', scope: .singleton}) app.refresh()! // → 自动调用 freeze_bean_definitions() // 冻结后 resolve() 走无锁快速路径 / Lock-free fast path after freeze svc := app.resolve('UserService')! // 查看缓存统计 / Check cache diagnostics println('Cache hits: ${app.cache_hit_count()}') println('Cache misses: ${app.cache_miss_count()}') println('Cache size: ${app.cache_size()}') println('Frozen: ${app.is_frozen_bean_definitions()}') // Spring 等价：DefaultListableBeanFactory.freezeConfiguration()
💡 性能提示：冻结后 resolve() 完全无锁，适合高并发热路径。缓存命中/未命中统计可用于诊断和调优。/ Performance tip: After freezing, resolve() is completely lock-free, ideal for hot paths. Cache hit/miss stats are useful for diagnostics and tuning.

EventBus 分片锁
ShardedRwMutex — 高并发事件分发优化

EventBus 从全局 RwMutex 升级到 ShardedRwMutex（16 分片），不同事件名的 dispatch()/on()/off() 操作可并发执行，大幅减少锁竞争。全局操作（如 off_listener()、shutdown()）使用 lock_all()/rlock_all()。

ShardedRwMutex API

API说明
sharded_mu.rlock(key)按 key 分片读锁 / Shard read lock by key
sharded_mu.runlock(key)按 key 分片读解锁 / Shard read unlock by key
sharded_mu.lock(key)按 key 分片写锁 / Shard write lock by key
sharded_mu.unlock(key)按 key 分片写解锁 / Shard write unlock by key
sharded_mu.rlock_all()全局读锁（遍历所有分片）/ Global read lock (all shards)
sharded_mu.runlock_all()全局读解锁 / Global read unlock
sharded_mu.lock_all()全局写锁（遍历所有分片）/ Global write lock (all shards)
sharded_mu.unlock_all()全局写解锁 / Global write unlock
v
// 不同事件名并发 dispatch — 互不阻塞 // Concurrent dispatch on different event names — no blocking spawn { bus.dispatch(core.new_event('user.registered', 'alice')) } spawn { bus.dispatch(core.new_event('order.created', 'order-123')) } // ↑ 两个事件映射到不同分片，可并行执行 // 全局操作：off_listener / shutdown 使用 lock_all bus.off_listener(listener_id) // → sharded_mu.lock_all() bus.shutdown() // → sharded_mu.lock_all() // Spring 等价：ConcurrentHashMap segment locking // Go 等价：sync.Map internal sharding

Core 锁设计说明
Lock Design — 锁分离策略与并发优化

resolve_cache 独立锁（cache_mu）

ApplicationContext 使用独立的 cache_mu（sync.RwMutex）保护 resolve_cache，与 bean_definitions 的 mu 分离。这样 resolve() 操作不会阻塞 register() 操作，反之亦然。

锁保护对象使用场景
mu (RwMutex)state, 应用状态refresh/start/stop/shutdown 状态变更
cache_mu (RwMutex)resolve_cache, frozen_bean_defs, is_frozen, cache_hits/missesresolve() 缓存读写、freeze/unfreeze
sharded_mu (ShardedRwMutex)listeners, transactional_listenersEventBus 分片锁：dispatch/on/off
v
// 锁分离示例：resolve 不阻塞 register // Lock separation: resolve does not block register // Goroutine A: resolve (uses cache_mu) svc := app.resolve('UserService')! // cache_mu.rlock() → 查缓存 → runlock // Goroutine B: register (uses container.mu) app.register(def)! // container.mu.lock() → 注册 → invalidate_cache() → unlock // A 和 B 不互相阻塞！ // A and B do NOT block each other!

EventBus 分片锁策略

EventBus 使用 16 分片 ShardedRwMutex，事件名通过 FNV-1a 哈希映射到分片。单事件名操作（dispatch/on/off）仅锁定对应分片；全局操作（off_listener/shutdown）锁定所有分片。

💡 设计原则：锁的粒度越细越好。resolve 缓存使用独立锁避免与注册操作竞争；EventBus 使用分片锁避免不同事件名之间的竞争。/ Design principle: Finer lock granularity is better. Resolve cache uses a separate lock to avoid contention with registration; EventBus uses sharded locks to avoid contention between different event names.

自动配置类在用户 Bean 注册之后处理，遵循"用户优先"原则。通过条件装配实现按需加载。

v
// 定义自动配置 @[auto_configuration] @[conditional_on_property('cache.enabled', 'true')] @[conditional_on_missing_bean('CacheManager')] pub struct CacheAutoConfiguration {} pub fn (c &CacheAutoConfiguration) configure(mut ctx core.ApplicationContext) ! { // 仅当 CacheManager 不存在时，注册默认实现 ctx.register(core.BeanDefinition{ type_name: 'CacheManager' scope: .singleton }) } pub fn (c &CacheAutoConfiguration) order() int { return 100 } // 注册到 AutoConfigurationManager mut app := core.new_application_context() app.auto_config_manager.add_candidate(core.AutoConfigurationCandidate{ type_name: 'CacheAutoConfiguration' config: &CacheAutoConfiguration{} conditions: [&OnPropertyCondition{key: 'cache.enabled', value: 'true'}] order_: 100 }) // refresh() 时自动处理 app.refresh()! // → 评估条件 → 注册 Bean → 调用 configure()
🚀 生产实践
  • 自动配置类保持轻量，只注册 Bean 定义，不做重逻辑
  • 使用 @[conditional_on_missing_bean] 让用户可以覆盖默认配置
  • order() 值越小越先处理，确保基础设施（DB、Cache）先于业务 Bean
  • 将自动配置集中到 auto_configuration.v 文件，便于管理和调试

便捷 DI 方法
register_component · create_and_wire · autowire_bean · get_service — 减少样板代码

Photon 提供一组泛型便捷函数，将常见的"注册 → 创建 → 注入 → 解析"流程封装为一行调用，避免手动构建 BeanDefinition 和处理 voidptr 类型转换。

autowire_bean[T](mut ctx, mut bean)!自动注入已有实例的 @[autowired] 字段

控制器依赖注入
autowire_controller · wire_controller — Spring @Controller + @Autowired 等价

Photon 提供编译期扫描控制器字段 @[autowired] 注解，并从 ApplicationContext 自动注入依赖的能力。这是实现 Spring 风格「控制器任意拆分」的关键 DI 基础设施。控制器可在任意目录、任意包中定义，只需在挂载点 import 即可。

方法返回值说明
ctx.autowire_controller[T](controller)!扫描 T 的 @[autowired] 字段并注入依赖
ctx.wire_controller[T]()!&T创建新实例 + 自动注入依赖（一步到位）
ctx.register_controller[T]()!&T注册控制器为单例 + 注入依赖，可通过 resolve() 再次获取
core.locate_controller[T]()!&T通过全局 ServiceLocator 创建并注入（无需传递 Context）
v
// 1. 定义控制器（可在任意文件/包中） @[controller] @[prefix: '/api/v1'] pub struct UserController { user_service &UserService @[autowired] auth_service &AuthService @[autowired] } // 2. 从 DI 容器创建并注入依赖 mut app := core.new_application_context() app.refresh()! // 方式一：wire_controller — 创建 + 注入（一步到位） ctrl := app.wire_controller[UserController]()! app.WebModule.mount_controller[UserController](ctrl, '/api/v1') // 方式二：register_controller — 注册为单例 + 注入 ctrl2 := app.register_controller[OrderController]()! // 后续可通过 ctx.resolve('OrderController') 再次获取 // 方式三：locate_controller — 全局定位器（无需传递 Context） ctrl3 := core.locate_controller[HealthController]()! // Spring 等价：@Controller + @Autowired 自动注册 // Laravel 等价：app()->make(Controller::class)
💡 解析顺序：autowire_controller[T]() 按 @[qualifier('name')] → 字段名 → 首字母大写形式 的顺序解析依赖。仅对标注 @[autowired] 的字段执行注入，非注解字段不受影响。
⚠️ V comptime 限制：由于 V 0.5.1 无法在 comptime 中获取引用类型字段的完整类型名，控制器 DI 使用 unsafe 指针写入方式设置字段值。仅对指针类型字段安全，不要在非指针字段上使用 @[autowired]。

AOP 切面支持
transactional_wrap · cacheable_wrap · detect_aop_methods — 编译期注解检测 + 辅助包装函数

V 语言无运行时反射，不支持运行时方法拦截（如 CGLIB / JDK 动态代理）。Photon 采用务实的 AOP 方案：编译期注解检测 + 辅助包装函数。

组件说明Spring 等价

cacheable_wrap[T](mut cache, key, ttl, f)缓存包装：先查缓存，miss 则执行并缓存@Cacheable advice

缓存包装示例

v
@[cacheable] pub fn (mut s UserService) get_user(id int) !string { return core.cacheable_wrap(mut s.cache, 'user:${id}', 300, fn () !string { // 缓存未命中时执行 — 如数据库查询 return 'user_data' })! }

限定符@[qualifier: 'redis-cache']@[qualifier('redis-cache')]
