RAII Guard 自动回收
PooledGuard[T] · 类型安全的 RAII 守卫 · 零心智成本

PooledGuard[T] 是类型安全的 RAII 守卫，当释放时（通过 defer 或手动），对象会自动归还到池中。线程安全的双重释放防护，防止意外归还。

API说明
PooledGuard[T]类型安全的 RAII 守卫，释放时自动归还对象到池，线程安全双重释放防护
acquire_guard[T]()获取池化对象并包装在 PooledGuard 中
with_acquired[T]()零心智成本回调 API：框架保证获取/释放，用户无需关心
v
import photon.pool mut p := pool.new_pool_with_config('db-pool', fn () !voidptr { conn := &DbConnection{host: 'localhost'} return unsafe { conn } }, 2, 10) p.initialize()! // Pattern 1: Guard with defer (recommended / 推荐模式) mut guard := pool.acquire_guard[DbConnection](mut p)! defer { guard.release() } conn := guard.get() conn.query('SELECT ...') // Even if query panics, defer ensures release / 即使 panic，defer 也会确保释放 // Pattern 2: Callback mode (zero cognitive load / 零心智成本) pool.with_acquired[DbConnection](mut p, fn (conn &DbConnection) ! { conn.query('SELECT ...') })! // Framework guarantees acquire/release — user never thinks about it // 框架保证获取/释放 — 用户完全不需要考虑

PoolAutoManager 跟踪应用内创建的所有池，确保应用关闭时全部正确关闭，防止池泄漏。

API说明

import photon.pool mut mgr := pool.new_pool_auto_manager() // Register pools / 注册池 mgr.register('db-pool', db_pool) mgr.register('redis-pool', redis_pool) // During application shutdown / 应用关闭时 mgr.close_all() // Closes all pools automatically / 自动关闭所有池 // Monitor all pools / 监控所有池 stats := mgr.pool_stats() for name, s in stats { println('${name}: active=${s.active}, idle=${s.idle}') } // Spring equivalent: DisposableBean + @PreDestroy on DataSource beans

RequestPool 请求对象池化
请求级对象复用 · 降低 GC 压力 · 优雅降级

RequestPool 管理请求级对象（Session、Input、MiddlewareContext 等）的池化复用，通过跨请求复用对象而非每次新建，降低高并发下的 GC 压力。池耗尽时自动创建临时对象（不阻塞请求）。

核心 API

API说明
new_request_pool(config)创建请求对象池管理器 / Create request pool manager
register_pool[T](name, factory, reset_fn)注册类型化子池 + 重置函数 / Register typed sub-pool with reset function
acquire[T](name)获取池化对象（返回 PooledGuard[T]）/ Acquire pooled object as RAII guard
release(name, obj)归还对象（自动调用 reset_fn）/ Release object (auto-calls reset_fn)
with_acquired[T](name, callback)零心智成本回调 API / Zero-cognitive-load callback API
pool_stats()获取所有子池统计信息 / Get stats for all sub-pools
close_all()关闭所有子池 / Close all sub-pools

Resettable 接口

池化对象在归还前必须调用 reset() 清除请求特定数据，防止跨请求泄漏。Photon 提供预置 reset 函数：reset_session()、reset_middleware_context()、reset_input()。

v
import photon.web // 1. 创建请求池 / Create request pool rp := web.new_request_pool(web.RequestPoolConfig{min_size: 4, max_size: 100}) // 2. 注册类型化子池 / Register typed sub-pool rp.register_pool[Session]('session', fn () !voidptr { return unsafe { voidptr(web.new_session('')) } }, web.reset_session)! // 3. RAII Guard 模式 / RAII Guard pattern mut guard := rp.acquire[Session]('session')! defer { guard.release() } sess := guard.get() sess.set('user_id', '42') // 4. 零心智成本回调 / Zero-cognitive-load callback rp.with_acquired[Session]('session', fn (sess &Session) ! { sess.set('user_id', '42') })! // 5. 监控 / Monitor stats := rp.pool_stats() for name, s in stats { println('${name}: active=${s.active}, idle=${s.idle}') } // Spring equivalent: @Scope("request") + ObjectPool
💡 优雅降级：当池耗尽时（所有对象在使用中且达 max_size），RequestPool 自动在池外创建临时对象，确保请求永不阻塞。临时对象在 release 时直接销毁而非归还到池。/ Graceful degradation: When the pool is exhausted, RequestPool creates temporary objects outside the pool, ensuring requests are never blocked. Temporary objects are destroyed on release instead of being returned to the pool.

Pool 锁设计说明
Lock Design — 分层锁策略与读多写少优化

RequestPool 分层锁策略

RequestPool 内部使用 RwMutex 保护 pools map。acquire 使用 rlock（读多写少场景），register_pool/close_all 使用 lock。rlock 仅保护 entries map 的读取，获取 entry 后立即释放锁，后续的 pool.acquire_guard[T]() 由 pool.Pool 自身的锁保护。这种分层锁策略避免了 RequestPool.mu 与 Pool 内部锁的嵌套。

操作锁类型说明
acquire[T]()rlock读锁：查找 entries map，获取后立即释放 / Read lock: lookup entries, release immediately
release()rlock读锁：查找 entries map + reset_fn / Read lock: lookup entries + reset_fn
register_pool[T]()lock写锁：新增子池 / Write lock: add sub-pool
close_all()rlock读锁：遍历关闭（不修改 map）/ Read lock: iterate and close
pool_stats()rlock读锁：遍历统计 / Read lock: iterate for stats
v
// 分层锁策略：RequestPool.rlock → Pool 内部锁 // Layered lock strategy: RequestPool.rlock → Pool's internal lock // Step 1: rlock 仅保护 entries map 读取 rp.mu.rlock() entry := rp.entries['session'] rp.mu.runlock() // 立即释放，不持有到 pool.acquire // Step 2: pool.acquire_guard 由 Pool 自身的 Mutex 保护 guard := entry.pool.acquire_guard[Session]() // 两把锁不嵌套 → 无死锁风险 / No lock nesting → no deadlock risk
🚀 生产实践
  • min_size 设置为常态并发量，减少冷启动时的连接创建开销
  • max_size 不超过数据库最大连接数 / 系统文件描述符上限
  • 设置 idle_timeout 避免空闲连接长期占用资源
  • 使用 PooledGuard[T] + defer 或 with_acquired[T]() 确保对象一定归还，零心智成本
  • 监控 pool.stats()，当 wait_count 持续增长时扩容
