编码器
JSON / Console 双格式
v
// JSON 编码器 — 生产环境推荐（便于 ELK/Grafana 解析） mut json_enc := logger.new_json_encoder(logger.EncoderConfig{ time_key: 'ts' level_key: 'level' message_key: 'msg' time_format: .rfc3339 }) log.set_encoder(json_enc) // 输出示例： // {"ts":"2025-01-15T10:30:00Z","level":"info","msg":"Server started"} // Console 编码器 — 开发环境（可读性强） mut console_enc := logger.new_console_encoder(logger.EncoderConfig{ time_format: .iso8601 }) log.set_encoder(console_enc) // 输出示例： // 2025-01-15 10:30:00 [INFO] Server started
🏷️
MDC 上下文
Mapped Diagnostic Context — 请求级追踪

MDC 允许在请求上下文中注入键值对，后续所有日志自动携带这些字段，非常适合请求追踪。

v
// 在中间件中注入 Request-ID log.put('request_id', 'req-abc123') log.put('user_id', '42') // 后续所有日志自动携带 request_id 和 user_id log.info('Processing order') // {"ts":"...","level":"info","msg":"Processing order","request_id":"req-abc123","user_id":"42"} // 请求结束后清理 log.remove('request_id') log.remove('user_id') // Spring 等价：MDC.put("requestId", "req-abc123") // Go Zap 等价：logger.With(zap.String("request_id", "req-abc123"))

时间格式
Unix / RFC3339 / ISO8601
格式示例输出适用场景
.unix1705312200日志分析系统（纯数字排序快）
.rfc33392025-01-15T10:30:00ZJSON 日志（标准格式）
.iso86012025-01-15 10:30:00Console 日志（人类可读）
🚀 生产实践
  • 生产环境使用 JSON 编码器 + RFC3339 时间，便于 ELK/Loki 采集分析
  • 通过 request_id_middleware 自动注入 Request-ID 到 MDC，实现全链路追踪
  • 敏感信息（密码、Token）在日志中自动脱敏
  • 设置合理的日志级别：开发 debug，生产 info
  • 热路径避免 debug 级别的字符串拼接，先检查 log.is_enabled(.debug)

LoggerFactory 日志工厂
命名空间分层 · 类型安全 · 热更新

LoggerFactory — 创建并管理 Logger 实例，支持基于命名空间的分层级别解析。Spring 等价：org.slf4j.LoggerFactory

方法说明Spring 等价
new_logger_factory(config)使用 LoggerConfig 创建工厂—
get_logger(name)创建或获取命名 Logger—
get_logger_for[T]()创建以类型 T 命名的 Logger（使用 T.name 作为命名空间）LoggerFactory.getLogger(MyClass.class)
reload_all()使用当前配置级别热更新所有缓存 LoggerLogback 的 JMX 级别重配置
logger_count()缓存 Logger 数量—
v
import photon.logger // 1. 创建 LoggerFactory mut factory := logger.new_logger_factory(config) // 2. 获取命名 Logger log_ := factory.get_logger('com.photon.UserService') // 3. 获取类型安全 Logger（推荐） log_ := logger.get_logger_for[UserService](mut factory) // 等价于：LoggerFactory.getLogger(UserService.class) // 4. 配置变更后热更新所有 Logger factory.reload_all() // 5. 检查缓存日志器 println('${factory.logger_count()} loggers cached') // Spring 等价：@Slf4j (Lombok) / LoggerFactory.getLogger(MyClass.class)

LogContext 上下文追踪
请求级结构化上下文 · trace_id / request_id 自动传播

LogContext — 请求级结构化日志上下文，提供 trace_id、request_id 等上下文字段的自动传播。灵感来自 Spring Sleuth 的追踪传播机制。

方法说明
new_log_context()创建空上下文
with_trace_id(id)设置 Trace ID（链式调用）
with_request_id(id)设置 Request ID（链式调用）
with_span(name)设置 Span 名称（链式调用）
with_field(key, value)添加自定义字段（链式调用）
to_fields()转换为 Map 供 Logger 使用
v
import photon.logger // 链式创建上下文 mut ctx := logger.new_log_context() .with_trace_id('trace-abc123') .with_request_id('req-456') .with_span('process_order') .with_field('user_id', '42') // 配合 Logger 使用 log_.with_fields(ctx.to_fields()) log_.info('Processing order') // Spring Sleuth 等价：MDC 中的 trace 传播 // Go 等价：context.Context with structured fields
📤
MultiOutputLogger 多输出
同时输出到多个目标 · Console + File + Remote

MultiOutputLogger — 包装 Logger 并同时将输出发送到多个目标（如控制台 + 文件）。Spring 等价：Logback 的 Appender 链。

v
import photon.logger mut multi := logger.new_multi_output_logger(my_logger) multi.add_output(fn (msg string) { // 写入文件 os.write_file('app.log', msg + '\n') or {} }) multi.add_output(fn (msg string) { // 发送到远程 http_post('https://logs.example.com/ingest', msg) or {} }) // 所有日志同时输出到原始 Logger + 所有额外输出 multi.info('Server started') multi.error('Connection failed') // Spring 等价：Logback 的 Appender 链
