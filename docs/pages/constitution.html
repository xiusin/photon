文档宪法

确保文档与代码永远同步的规则体系 — 当功能变更时，相关文档必须同步更新。这是所有开发者的强制性义务。

📜 宪法强制执行文档即代码
📜
前言
为什么需要文档宪法？
🏛️ 核心原则

过时的文档比没有文档更危险。它误导开发者、浪费排查时间、损害框架信誉。本宪法确立"文档即代码"的原则：文档与源码享有同等的版本管理、Code Review 和持续集成地位。

⚖️
宪法条款
9 条强制规则

1.1 新增功能

新增注解、API、模块或行为时，必须同步更新相关 HTML 页面：

  • 新增 @[annotation] → 更新 core.html 的注解表 + 对应模块页面的用法示例
  • 新增 public 函数 → 更新对应模块页面的 API 参考表
  • 新增模块 → 更新 index.html 的模块索引 + 创建新模块 HTML 页面

2.1 API 变更

任何 public API 的变更（签名、行为、废弃）必须同步更新文档：

  • 函数签名变更 → 更新 API 参考表中的参数和返回值
  • 行为变更 → 更新用法示例和说明文字
  • API 废弃 → 在文档中标注 @deprecated 和迁移路径

3.1 代码示例验证

文档中的所有代码示例必须：

  • 与当前 API 兼容（不引用已删除的函数/类型）
  • 可编译运行（不包含省略号占位符 ... 除非明确标注为伪代码）
  • 遵循 V 语言官方风格指南

4.1 Spring/Laravel 类比同步

comparison-grid 和 info-spring/info-laravel 中的类比必须准确：

  • 新功能添加类比时，需确认 Spring/Laravel 中确实存在对应概念
  • 类比不是等价 — 必须注明差异点（如有）
  • Spring Boot 版本以 3.x 为准，Laravel 版本以 11.x 为准

5.1 生产实践同步

production.html 和各模块的 production-box 必须反映当前安全最佳实践：

  • 新增安全相关功能 → 更新 production.html 对应章节
  • 性能相关变更 → 更新性能优化建议
  • 部署方式变更 → 更新部署指南

6.1 版本标记

每个模块页面必须标注当前成熟度徽章：

  • GA — 生产就绪，API 稳定
  • Beta — 功能完整，可能微调
  • Alpha — 实验性，API 可能大幅变更

成熟度变更时，同步更新 index.html 模块索引和模块页面的 hero-tags。

7.1 文档结构一致性

所有模块页面遵循统一结构：

结构模板
1. Hero（标题 + 简介 + 标签） 2. 快速上手（最简可用示例） 3. 核心概念（接口/结构体说明） 4. 详细用法（分场景 Demo） 5. API 参考（方法列表 + 参数表） 6. Spring/Laravel 对比 7. 🚀 生产实践（production-box） 8. 页面导航（上一页 / 下一页）

8.1 Code Review 检查项

PR 审查时必须检查：

  • 源码变更是否涉及 public API？→ 对应文档是否同步更新？
  • 新增的 @[annotation] 是否在文档中有说明？
  • 代码示例是否可编译运行？
  • production-box 中的建议是否仍然适用？
  • 成熟度徽章是否需要更新？

9.1 违宪处理

发现文档与代码不一致时：

  • 立即创建 Issue 标记为 docs:out-of-sync
  • 24 小时内提交修复 PR
  • 同一文件连续 3 次违宪 → 添加 CI 检查自动验证
